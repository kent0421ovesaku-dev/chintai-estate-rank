# 1. プロジェクト概要

## 1.1 背景
日本の上場企業は、有価証券報告書の注記として「賃貸等不動産」の帳簿価額や時価を開示している場合がある。これらの情報を集約すると、企業の貸借対照表上では見えにくい「不動産含み益」を横断比較できる。

一方で、この情報は企業ごとの有報に分散しており、投資家が一覧で確認するのは手間が大きい。そこで、EDINET API と iXBRL 解析を用いて、上場企業の賃貸等不動産データを収集・構造化し、検索・ランキング可能な形で公開する。

## 1.2 目的
本プロジェクトの目的は以下の通り。

1. 上場企業の賃貸等不動産データを一覧化する
2. 含み益、簿価、時価などを横断比較できるようにする
3. 投資家が割安株や資産株を発見しやすくする
4. 将来的には実質PBRや時系列推移を加え、分析メディアとして発展させる

## 1.3 サービスの基本方針
- フロントエンドは静的HTMLを基本とし、表示速度と運用の軽さを優先する
- データ生成は Python スクリプトで行い、最終成果物として \`output/index.html\` を出力する
- サーバーサイドの動的アプリケーションは当面採用しない
- まずは「一覧性」「比較性」「見つけやすさ」を重視し、凝った機能より正確性と実用性を優先する

## 1.4 想定ユーザー
### 主対象
- 日本株投資家
- バリュー株・資産株投資家
- 不動産含み益に着目する個人投資家
- 実質PBRや資産効率を重視する分析者

### 副対象
- アクティビスト動向を追う投資家
- 証券・IR・金融メディア関係者
- 企業分析コンテンツ制作者

## F-01: EDINET からの対象企業データ取得

### 概要
対象企業の有価証券報告書を EDINET 経由で取得する。

### 目的
賃貸等不動産注記を含む元データを自動取得するため。

### 要件
- 上場企業を対象に有価証券報告書を取得できること
- 対象書類種別を有価証券報告書に限定できること
- 対象決算期を指定できること
- 対象企業一覧をループ処理できること
- 取得失敗時にスキップまたは再試行できること
- 取得成否をログ出力できること

### 入力
- 証券コードまたはEDINETコード
- 対象年度
- 書類種別条件

### 出力
- iXBRLファイルまたは解析可能な取得結果
- 取得ログ

### 受け入れ条件
- 対象企業群に対して処理が停止せず最後まで走ること
- 失敗企業があっても処理全体が落ちないこと
- 後で再確認できるよう、失敗企業一覧がわかること

### 実装優先度
高

### 現状
概ね実装済み。ただし取得失敗時のロギングや再試行戦略は明文化不足。

## F-02: iXBRL から賃貸等不動産注記を抽出

### 概要
取得した有報の iXBRL から、賃貸等不動産に関する注記を解析する。

### 目的
簿価・時価・差額を機械的に取り出すため。

### 要件
- 賃貸等不動産注記の有無を判定できること
- 当期末帳簿価額を抽出できること
- 当期末時価を抽出できること
- 必要に応じて前期情報も扱えること
- 企業ごとに開示形式が多少異なっても、一定範囲まで耐性を持つこと
- 値の単位(百万円、千円など)を統一換算できること
- 欠損値を明示的に扱えること

### 入力
- iXBRLデータ

### 出力
- \`book_value\`
- \`market_value\`
- \`unrealized_gain = market_value - book_value\`
- 注記の有無フラグ
- 必要に応じて注記テキストの一部

### 受け入れ条件
- 少なくとも主要企業群で帳簿価額・時価・含み益を正しく抽出できること
- 単位換算ミスが発生しないこと
- 値が取れない場合に誤って0を入れず、欠損として扱うこと

### 実装優先度
最重要

### 現状
実装済みだが、全件検証は未完了。異常値検出とスポット再確認の仕組みが必要。

# 5. 非機能要件

## NF-01: パフォーマンス

### 要件
- 初回表示速度をできるだけ速くすること
- 静的HTMLとして配信し、不要なライブラリ依存を避けること
- 大量データ表示時でも主要端末で閲覧可能であること
- 検索・ソート応答が実用範囲内であること

### 目標
- 一般的なPC環境で数秒以内に表示
- モバイルでも致命的なフリーズを起こさない
- 可能ならHTMLサイズ削減やJS処理最適化を検討する

### 補足
現在 \`output/index.html\` は大容量化しているため、今後は以下も検討余地あり。
- 行データの分離
- JSON化
- 遅延描画
- ページ分割

---

## NF-02: データ正確性

### 要件
- 抽出元は原則として有価証券報告書の賃貸等不動産注記とする
- 単位換算ミスを防止する
- 欠損とゼロを区別する
- 異常値検出を導入する
- 看板銘柄や上位企業は優先的にスポット確認する

### 目標
- TOP10〜TOP30のデータ信頼性を重点的に担保する
- 全件完全手作業検証ではなく、異常値ベースで効率確認する

---

## NF-03: 可用性・堅牢性

### 要件
- 一部企業の取得失敗で全体処理が停止しないこと
- 外部API取得失敗時に例外処理があること
- 再実行で復旧できること
- 欠損があってもHTML生成が最後まで完走すること

---

## NF-04: 保守性

### 要件
- データ取得、解析、整形、HTML生成の責務を分けること
- ハードコードを減らすこと
- 設定値を定数化すること
- 業種マッピングやURLマスタを差し替えやすくすること

# 6. データ仕様

## 6.1 企業レコードの基本項目
各企業データは最低限以下を持つこと。

\`\`\`json
{
  "security_code": "0000",
  "company_name": "企業名",
  "industry": "業種",
  "book_value": 0,
  "market_value": 0,
  "unrealized_gain": 0,
  "unrealized_gain_ratio": 0,
  "company_website_url": "",
  "has_rental_real_estate_note": true,
  "source_fiscal_year": "2025",
  "source_document_type": "有価証券報告書"
}
\`\`\`

## 6.2 各項目定義

### \`security_code\`
- 型: string
- 内容: 証券コード
- 必須: 必須

### \`company_name\`
- 型: string
- 内容: 企業名
- 必須: 必須

### \`industry\`
- 型: string
- 内容: 業種名
- 必須: できれば必須

### \`book_value\`
- 型: number or null
- 内容: 賃貸等不動産の帳簿価額
- 単位: 統一された数値単位
- 必須: 任意

### \`market_value\`
- 型: number or null
- 内容: 賃貸等不動産の時価
- 必須: 任意

### \`unrealized_gain\`
- 型: number or null
- 内容: \`market_value - book_value\`
- 必須: 任意

### \`unrealized_gain_ratio\`
- 型: number or null
- 内容: 含み益率または任意定義指標
- 必須: 任意

### \`company_website_url\`
- 型: string
- 内容: 企業公式サイトURL
- 必須: 任意

### \`has_rental_real_estate_note\`
- 型: boolean
- 内容: 賃貸等不動産注記の有無
- 必須: 必須

### \`source_fiscal_year\`
- 型: string
- 内容: 対象決算年度
- 必須: 必須
